## Вступление

Данное руководство поможет быстро включиться в процесс разработки, правильно оформлять изменения и соблюдать единые стандарты.

---

## Сообщение об ошибках / Предложение новых возможностей

1. **Проверьте, нет ли уже похожего issue** – используйте поиск.
2. Если вы нашли ошибку или хотите предложить новую функциональность, **создайте новый issue**:
   - Выберите соответствующий шаблон (bug / feature request), если он есть.
   - Опишите проблему: шаги воспроизведения, ожидаемое и фактическое поведение, окружение (ОС, версия ПО).
   - Для предложений: объясните, зачем это нужно и как может быть реализовано.

---

## Как вносить изменения (Pull Request)

1. **Создайте ветку** от `main` (или `develop`) с понятным названием:
   `feature/краткое-описание`, `fix/краткое-описание`, `docs/...` и т.п.
2. **Внесите изменения**.
3. **Следуйте соглашению о коммитах** (см. ниже).
4. **Убедитесь, что все тесты проходят** и код соответствует стилю проекта.
5. **Создайте Pull Request** в основную ветку.
   - Заполните описание PR: укажите, что было сделано, ссылку на issue (если есть).
6. Дождитесь ревью – исправьте замечания, если они есть.

## Соглашение о коммитах (Conventional Commits)

Все коммиты **обязаны** соответствовать спецификации [Conventional Commits 1.0.0](https://www.conventionalcommits.org/).
Это позволяет автоматически генерировать CHANGELOG и правильно вычислять версию.

### Структура коммита

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

- **type** – тип изменения (обязательно).
- **scope** – область, к которой относится изменение (опционально, в круглых скобках).
- **description** – краткое описание (до 72 символов).
- **body** – более подробное описание (через пустую строку после description).
- **footer** – дополнительная информация (через пустую строку после body), например, ссылки на задачи, `BREAKING CHANGE`.

### Разрешённые типы


| Тип   | Описание                                                                                |
| ---------- | ------------------------------------------------------------------------------------------------- |
| `feat`   | Новая функциональность (MINOR-релиз в SemVer)                        |
| `fix`    | Исправление ошибки (PATCH-релиз в SemVer)                                |
| `docs`   | Изменения в документации                                                  |
| `style`  | Форматирование кода, опечатки, без изменения логики |
| `test`   | Добавление или исправление тестов                                 |
| `build`  | Изменения в системе сборки или зависимостях               |
| `ci`     | Изменения в CI/CD конфигурации                                            |
| `revert` | Откат предыдущего коммита                                                |

*Примечание:* типы `feat` и `fix` влияют на версию. Остальные типы — нет, если только они не содержат `BREAKING CHANGE`.

### Scope (область)

Scope указывает модуль, компонент или часть проекта, которую затрагивает изменение.
Примеры: `api`, `ui`, `auth`, `core`, `docs`.
Вы можете использовать свои области, но старайтесь быть последовательными.

### BREAKING CHANGE (ломающие изменения)

Если изменение **ломает обратную совместимость**, его необходимо пометить одним из способов:

1. **Восклицательный знак `!` после типа/scope**
   ```
   feat(api)!: изменить формат ответа
   ```
2. **Футер `BREAKING CHANGE:`**
   ```
   fix: удалить устаревший метод

   BREAKING CHANGE: метод `oldMethod` удалён, используйте `newMethod`.
   ```

Любой коммит с `!` или `BREAKING CHANGE` в футере будет считаться **MAJOR**-релизом (независимо от типа).

### Примеры коммитов

```
refactor(api): улучшить обработку ошибок

- заменить `throw` на возврат `Result`
- добавить логирование

Closes #42
```

### Проверка коммитов

В проекте настроен **commitlint** и **husky**, которые автоматически проверяют сообщение коммита перед сохранением.
Если коммит не соответствует правилам, вы увидите ошибку и сможете исправить сообщение.

## Процесс разработки

1. **Клонируйте репозиторий**
   `git clone https://github.com/ваш-проект.git`
2. **Установите зависимости**
   `npm install` (или `yarn`, `pnpm` в зависимости от проекта)
3. **Создайте ветку** от `main`
   `git checkout -b feature/ваша-фича`
4. **Вносите изменения**, соблюдая:

   - Кодстайл (линтеры запускаются автоматически перед коммитом через husky/lint-staged).
   - Наличие тестов для новой функциональности или исправлений.
   - Обновление документации, если нужно.
5. **Запустите тесты локально**
   `npm test`
6. **Сделайте коммит** с корректным сообщением.
7. **Отправьте ветку** в удалённый репозиторий и создайте Pull Request.

---

## Pull Request

- В заголовке PR кратко опишите суть изменений.
- В описании:
  - Укажите, что именно было сделано.
  - Ссылку на issue (например, `Closes #123`).
  - Если изменение ломающее, отметьте это (`BREAKING CHANGE`).
- Дождитесь проверки CI (тесты, линтеры).
- После ревью внесите правки (если нужно) и запросите повторную проверку.

---

## Кодстайл и линтеры

- Мы используем **ESLint** (для JavaScript/TypeScript) / **Prettier** (для форматирования). Конфигурация находится в корне проекта.
- Запустить проверку вручную: `npm run lint`
- Автоисправление: `npm run lint:fix`
